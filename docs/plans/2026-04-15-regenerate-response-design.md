# 重新生成最后一条回答设计

## 目标

为聊天区最后一条已完成的 `assistant` 回复增加“重新生成”能力。用户点击后，系统基于“上次用户提问的上下文”重新发起一次真实请求，并在原位置重新生成回答。

## 范围

### 本功能做什么

- 仅对最后一条、已完成的 `assistant` 回复提供“重新生成”按钮
- 点击后立即清空该回复内容，并像首次回答一样进入新的流式生成状态
- 复用上次用户提问的上下文重新回答
- 使用当前最新的运行配置执行本次重新生成
- 新结果直接写回原 `assistant` 消息槽位，不新增可见消息

### 本功能不做什么

- 不支持对任意历史 `assistant` 回复重新生成
- 不保留旧回复用于对比
- 不提供失败回滚
- 不单独为重新生成设计新的错误展示形态
- 不绕开正常工具执行；如果回答涉及工具调用或文件改动，重新生成时仍按正常流程再次执行

## 用户确认过的产品定义

### 重新生成的准确语义

- “重新生成”表示：使用上次用户提问的上下文，重新回答
- 除了这份提问上下文之外，其它执行条件都读取当前最新设置

### 提问上下文包含

- 上次用户消息文本
- 上次用户消息携带的附件
- 上次用户消息里的文件标签 / 路径引用
- 任何已经实际进入该用户消息载荷的上下文内容

### 运行配置不复用，实时读取

- 当前 provider
- 当前 model
- 当前 permission mode
- 当前 reasoning effort
- 当前 Agent
- 其它会影响执行的当前运行时设置

## 交互设计

### 按钮显示规则

仅当同时满足以下条件时显示“重新生成”按钮：

- 当前消息列表最后一条消息是 `assistant`
- 该消息已完成，非 `streaming`
- 当前没有进行中的生成请求
- 能可靠拿到上一条用户提问上下文

以下情况不显示或禁用按钮：

- 当前正在 `loading` 或 `streamingActive`
- 最后一条消息不是 `assistant`
- 缺少可重用的上一条用户提问上下文

### 点击后的界面行为

点击“重新生成”后，界面表现必须与“首次回答”一致：

1. 立即清空最后一条 `assistant` 回复内容
2. 将该消息切换到新的流式生成状态
3. 使用“旧提问上下文 + 当前最新运行配置”发起真实请求
4. 新的流式文本、thinking、工具块和最终结果直接写回这条消息
5. 如果失败，沿用首次回答失败时的现有表现，不恢复旧内容

### 为什么不保留旧结果

本设计明确不保留旧结果，也不提供回滚。原因是：

- 用户要求交互与首次回答完全一致
- 增加旧结果备份和回滚会扩大状态复杂度
- 额外分支会抬高引入新 bug 的概率

## 技术设计

### 总体策略

推荐实现路径：覆盖最后一条 `assistant` 消息槽位，重新跑一轮真实请求。

关键约束：

- 不新增可见 `user` 消息
- 不新增可见 `assistant` 消息
- 直接复用现有最后一条 `assistant` 的槽位
- 重新启动一轮标准 streaming 生命周期

### 前端新增状态

#### 1. `lastReplayablePromptContextRef`

用于保存最近一次成功提交的“用户提问上下文”，仅保存会被重新使用的输入内容。

建议字段：

```ts
interface ReplayablePromptContext {
  text: string;
  attachments?: Attachment[];
  fileTagsInfo?: Array<{
    displayPath: string;
    absolutePath: string;
  }> | null;
}
```

该结构不包含 provider、model、permissionMode、reasoningEffort、Agent 等运行配置。

#### 2. `regenerationState`

用于标识当前是否处于重新生成流程，以及当前要复用的目标消息。

建议字段：

```ts
interface RegenerationState {
  active: boolean;
  targetMessageKey: string | null;
}
```

不需要：

- `backupMessage`
- 临时缓冲区
- 成功替换 / 失败回滚双分支

### 正常发送时需要补的记录

在用户正常发送消息、并真正调用发送链路前，记录可重放的提问上下文：

- `text`
- `attachments`
- `fileTagsInfo`

这样“重新生成”拿到的是最近一次真实发送过的用户提问上下文，而不是从界面文本反推。

### 重新生成时的发送链路

点击“重新生成”后：

1. 校验最后一条消息是否为可重生成的 `assistant`
2. 校验 `lastReplayablePromptContextRef` 是否存在
3. 从当前最新状态读取运行配置：
   - `provider`
   - `model`
   - `permissionMode`
   - `reasoningEffort`
   - `Agent`
4. 立即清空最后一条 `assistant` 消息内容，并标记为新的 streaming 目标
5. 使用“旧提问上下文 + 当前最新配置”再次走现有发送链路
6. 新流式结果直接写回这条 `assistant` 消息

### 为什么读取最新运行配置

不复用旧运行配置，是为了避免以下问题：

- 用户切换了 provider，旧 provider 已不可用
- 用户切换了 model，旧 model 已失效
- 用户切换了 Agent，希望新回答遵从当前 Agent

因此本功能的语义是：

- 问题不变
- 运行环境按当前状态重新取

### 对现有流式机制的要求

当前前端已经以“最后一条 `assistant` + `streamingActive` + `__turnId`”为主线维护流式消息。重新生成应尽量复用这条机制：

- 重新生成开始时，把目标消息重新置为流式消息
- 重新生成的内容更新继续通过现有 streaming 更新逻辑落到目标消息
- 重新生成结束时，按现有完成逻辑收口

不建议单独发明一套“重新生成专用消息渲染逻辑”。

## 错误处理

重新生成失败时，行为必须与首次回答失败保持一致：

- 不回滚旧内容
- 不显示新的专属错误形态
- 直接复用当前发送/流式失败后的提示与消息表现

典型失败来源：

- 当前 provider / model / Agent 配置不可用
- 工具执行失败
- 网络或桥接异常
- 权限或执行链路报错

## 边界条件

- 仅最后一条 `assistant` 可重新生成
- 当前有进行中的生成请求时，不允许触发重新生成
- 如果最后一条不是 `assistant`，不显示按钮
- 如果无法恢复上一条用户提问上下文，不显示按钮
- 重新生成期间不允许再次点击“重新生成”
- 含工具调用、命令执行、文件修改的回答同样允许重新生成，并按正常流程再次执行

## 测试方案

### 前端单测

- 只有最后一条已完成 `assistant` 显示“重新生成”按钮
- 当前 `loading` / `streamingActive` 时按钮隐藏或禁用
- 点击后最后一条 `assistant` 被立即清空并进入 streaming 状态
- 重新生成使用的是“旧提问上下文 + 当前最新运行配置”
- 切换 provider / model / Agent 后重新生成，请求参数使用最新设置
- 失败时不回滚，沿用当前失败展示

### 链路测试

- 纯文本回答重新生成
- 包含工具调用的回答重新生成
- 包含文件修改的回答重新生成
- 切换模型后重新生成
- 切换 Agent 后重新生成

### 手工验收

1. 正常发起一轮问答
2. 在最后一条 `assistant` 上点击“重新生成”
3. 验证旧内容是否立即清空
4. 验证界面是否像首次回答一样流式生成
5. 验证成功结果是否落在原位置
6. 验证失败时是否没有回滚，且表现与首次失败一致

## 风险与控制

### 主要风险

- 误把“重新生成”做成“新增一轮消息”，导致历史结构脏化
- 在 streaming 生命周期里错误复用了旧 `assistant` 状态，导致工具块或文本残留
- 混用了旧运行配置和新运行配置，导致行为不一致

### 控制策略

- 严格限定为复用最后一条 `assistant` 槽位
- 严格区分“用户提问上下文”和“运行配置”
- 尽量复用现有 streaming 逻辑，不新造并行消息模型

## 实现建议

优先落地顺序：

1. 补“上一条用户提问上下文”的记录结构
2. 为 `MessageItem` / `MessageList` 增加“重新生成”按钮显示逻辑
3. 实现点击后的消息清空与重新发送
4. 接入现有 streaming 更新逻辑
5. 补单测和手工验收
